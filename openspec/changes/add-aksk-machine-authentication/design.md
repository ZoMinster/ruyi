## Context

当前宿主认证中间件只识别`Authorization: Bearer <JWT>`，认证成功后把用户、会话和租户信息写入`bizctx`，权限中间件再从用户角色访问快照判断`g.Meta permission`。该模型适合交互式用户会话，但无法表达没有用户身份、没有在线会话且权限独立管理的外部机器调用方。

现有接口权限字符串也不是接口唯一标识。例如多个列表、详情和候选接口可能共享同一个`permission`。机器访问既要精确到接口，又要对接口所操作的资源类型区分整体`read`与`write`，因此不能直接把`AK`绑定到现有用户、角色或菜单权限，也不能把机器身份伪装成`UserId`进入当前用户授权链。

本次变更跨越宿主认证、插件公共契约、路由元数据、接口文档、缓存协调、操作日志和新源码插件。用户已明确授权修改`apps/lina-core`，但核心修改必须保持通用认证和授权接缝，不得把`linapro-auth-aksk`的表结构、管理页面或`HMAC`细节写入宿主核心领域。

## Goals / Non-Goals

**Goals:**

- 在宿主建立可承载`user`与`machine`的通用认证主体模型，并保持现有`JWT`用户行为。
- 允许源码插件通过稳定提供者接缝注册新的认证方案和机器授权实现。
- 让机器访问同时满足精确接口授权和资源类型级`read`、`write`授权。
- 由`linapro-auth-aksk`闭环管理机器客户端、多个访问密钥、策略、签名校验、页面和插件数据。
- 保证密钥存储、重放防护、权限变更、插件启停和集群多节点场景按失败关闭工作。
- 让静态接口、源码插件接口和动态插件接口使用同一套机器访问元数据与授权判定语义。
- 在设计阶段明确数据库访问次数、缓存失效、审计脱敏、`i18n`和测试门禁。

**Non-Goals:**

- 不使用`AK/SK`替代用户登录、刷新令牌、在线会话或用户角色权限。
- 不允许机器客户端调用未显式开放给`machine`主体的接口。
- 不支持按资源`ID`、路径模式、字段条件、标签或任意表达式授权。
- 不引入显式`deny`、策略优先级、通用策略语言、`OPA`或逐资源`Casbin`检查。
- 不允许一个机器客户端或访问密钥跨平台与租户边界复用。
- 不把第三方云厂商`AK/SK`配置迁移到本模块，也不改变现有存储插件配置语义。
- 不在本次变更中增加请求配额、计费、来源`IP`白名单或客户端证书认证。

## Decisions

### 1. 宿主只拥有通用认证主体和提供者接缝

在`apps/lina-core/pkg/plugin/capability/authcap`下新增中立的认证主体和值对象，并在职责明确的`SPI`子包中定义认证提供者契约。宿主认证分派器、提供者生命周期和请求上下文适配由`apps/lina-core`实现；`linapro-auth-aksk`通过源码插件注册入口提供`LINA-HMAC-SHA256`实现。

通用契约至少表达：

```go
type Actor struct {
    Kind         ActorKind
    SubjectID    string
    CredentialID string
    TenantID     int
}

type AuthorizationRequest struct {
    OperationCode string
    ResourceCode  string
    AccessMode    AccessMode
}
```

`ActorKind`至少包含`user`和`machine`。机器认证结果不得设置`UserID`、不得创建在线会话、不得签发宿主用户`JWT`，也不得复用用户角色快照。`bizctx`和插件可见`CapabilityContext`增加机器主体投影，但保留用户字段的原有含义。

认证提供者通过构造函数注入共享运行期依赖并在启动装配阶段注册。请求路径不得临时`New()`提供者、缓存、协调服务或密钥仓库。新增依赖的 owner、创建位置、传递路径和共享实例策略必须在任务记录中追溯。

**备选方案：把`HMAC`直接写入宿主认证服务。** 未采用，因为这会让核心宿主长期拥有插件特定配置、表和协议实现，破坏通用宿主边界。

**备选方案：插件全局中间件把`AK/SK`转换成用户`JWT`。** 未采用，因为现有公开能力不能安全签发机器令牌，而且该做法会丢失机器主体、独立权限和审计语义。

### 2. 认证方案按前缀精确分派

宿主先解析`Authorization`方案，再选择唯一提供者：

- `Bearer`由现有`JWT`认证器处理。
- `LINA-HMAC-SHA256`由启用状态下的`linapro-auth-aksk`提供者处理。
- 缺少、未知、重复注册或格式非法的方案返回`401`。
- 已匹配提供者返回认证失败时立即返回`401`，不得回退到其他提供者。
- 认证成功但接口或资源权限不足时返回`403`。

插件禁用、卸载或提供者初始化失败时，`LINA-HMAC-SHA256`不可用并失败关闭；宿主和`Bearer JWT`继续工作。提供者注册名必须唯一，重复方案在启动装配阶段返回错误。缺少主密钥属于插件认证提供者不可用，不得阻断与机器认证无关的宿主启动、JWT管理路由或已有非密钥管理操作。

### 3. 接口使用稳定机器访问元数据

为允许机器访问的请求 DTO 或动态路由声明增加：

| 元数据 | 语义 |
|---|---|
| `operation` | 全局唯一且稳定的接口操作码 |
| `resource` | 受操作的稳定资源类型码 |
| `action` | `read`或`write` |
| `actors` | 允许的主体集合，例如`user,machine` |

未声明`actors`的现有受保护接口保持用户访问语义，但机器访问默认拒绝。声明包含`machine`时，`operation`、`resource`和`action`均为必填；重复`operation`、未知`action`或缺失资源码在启动路由审计、动态插件安装或启用校验中失败。

列表、详情、下载和导出归类为`read`；创建、更新、删除、导入、状态切换、触发和执行归类为`write`。接口仍保留现有`permission`供用户角色授权使用，机器授权不读取该字段。

**备选方案：用`HTTP method + path`作为机器权限键。** 未采用，因为路径版本或参数命名变化会无意改变授权身份，且一个权限码覆盖多个接口的现状无法表达稳定接口授权。

### 4. 机器授权采用接口与资源双重允许集合

机器请求必须同时满足：

```text
operationCode in allowedOperations
AND
resourceAccess[resourceCode][accessMode] == true
```

策略只有允许语义，多个策略取并集，默认拒绝，不支持显式`deny`和优先级。资源授权只判断资源类型整体`read`或`write`，不解析目标资源`ID`或路径模式。

策略快照在一次读取中装配为操作码集合和资源权限映射，请求判定使用常数时间内存查找。禁止在单次请求中按操作、策略或资源循环查询数据库。插件启停导致某接口不可用时，策略关系保留但不生效；重新启用后按最新路由目录恢复。

### 5. 机器客户端与访问密钥分离

一个机器客户端是稳定认证主体和策略绑定对象，可以拥有多个访问密钥。插件持久化模型至少包含：

| 表 | 主要职责 |
|---|---|
| `plugin_linapro_auth_aksk_machine_client` | 客户端身份、租户边界、状态和审计字段 |
| `plugin_linapro_auth_aksk_access_key` | `AK`、加密`SK`、状态、有效期和使用时间 |
| `plugin_linapro_auth_aksk_policy` | 策略名称、状态和租户边界 |
| `plugin_linapro_auth_aksk_policy_operation` | 策略允许的精确操作码集合 |
| `plugin_linapro_auth_aksk_policy_resource` | 策略允许的资源类型`read`、`write`位 |
| `plugin_linapro_auth_aksk_client_policy` | 客户端与策略关联 |

所有表使用完整插件前缀、数据库自动时间维护和`deleted_at`软删除。关联表使用业务唯一约束和集合化读写。常用认证查询至少具备未删除`AK`唯一索引；租户列表、客户端密钥列表、策略关联和策略资源装配按真实过滤路径设计组合索引。

访问密钥不提供修改`SK`接口。轮换通过为同一客户端创建第二把密钥、完成调用方切换后停用并删除旧密钥实现。删除立即使密钥不可认证，同时保留软删除元数据供审计关联。

### 6. `HMAC`协议和密钥存储采用版本化安全设计

请求使用以下头部：

```text
Authorization: LINA-HMAC-SHA256 Credential=<AK>,Signature=<hex>
X-Lina-Date: <Unix seconds>
X-Lina-Nonce: <base64url random value>
X-Lina-Content-SHA256: <lowercase hex SHA-256>
```

签名串固定包含协议版本、请求方法、规范化转义路径、按键和值排序并进行`RFC 3986`编码的查询参数、请求体摘要、时间戳和`nonce`。服务端复算请求体摘要、在恒定时间内比较签名，并默认只接受服务器时间前后`300`秒内的请求。协议规范和测试向量必须作为插件文档与自动化测试的一部分维护。

`SK`使用密码学安全随机源生成，只在创建密钥响应中返回一次。服务端以`AES-256-GCM`和随机`nonce`加密保存，关联数据至少包含租户、客户端和`AK`，避免密文被复制到其他主体。主密钥通过部署环境中的版本化密钥环注入，数据库只保存密钥版本、加密`nonce`和密文；缺少活动主密钥或历史密钥版本时，机器认证和新访问密钥创建必须失败关闭，但不得终止宿主或现有`Bearer JWT`管理面。

不能只存`SK`单向摘要，因为`HMAC`验证需要服务端持有可用于复算签名的对称密钥材料。解密后的`SK`只在当前认证调用中存活，不进入日志、错误、缓存或审计载荷；认证缓存只允许保存密文和非敏感策略快照。

### 7. 平台与租户边界独立于资源读写权限

机器客户端创建时固定为平台范围或当前单一租户范围，访问密钥继承客户端边界且不能单独修改。机器认证成功后由宿主写入不可伪造的`TenantID`，请求头或查询参数不得覆盖该值。

机器资源权限只决定资源类型整体读写，不替代资源模块原有租户过滤。机器可访问接口必须继续在数据库查询阶段按机器主体租户过滤，详情和写操作必须在执行前验证目标资源租户；聚合不得泄露其他租户数据。

机器客户端、密钥和策略属于平台或租户共享治理资源，不是某个管理员的私有业务数据。因此管理接口采用以下明确数据权限例外：`JWT`管理权限决定能否管理，平台或当前租户边界决定可见集合，不应用本人或部门数据范围。创建后同一治理范围内具有管理权限的管理员均可见；跨租户始终拒绝。该权威边界、拒绝策略和测试必须在任务及审查结论中记录。

### 8. 权限快照和重放状态使用统一协调基础设施

权威数据源是插件数据库表。认证读路径按`tenant + AK`读取版本化凭证与策略快照：

- 单机模式可以使用共享进程内读缓存和本地`nonce`去重，但必须复用宿主修订号抽象。
- 集群模式必须使用现有协调后端的共享修订号、事件或共享缓存完成跨节点失效。
- 密钥、客户端、策略或关联关系写入成功后，必须在事务提交后推进`machine-access`修订号；失效发布失败不得返回完全成功。
- 快照新鲜度无法确认且超过允许陈旧窗口时失败关闭，不得使用可能已撤销的权限放行。
- `nonce`在认证通过前通过共享原子“不存在则写入”语义登记，`TTL`至少覆盖签名时间窗口；同一`AK + nonce`重复出现时拒绝。
- 集群协调后端不可用时，`AK/SK`认证失败关闭，不退化为节点本地重放状态；`JWT`是否可用继续遵守其现有契约。

`last_used_at`不是认证权威数据。实现按每把密钥固定时间窗口合并更新，允许短暂陈旧，避免每个请求产生数据库写入；更新时间写入失败记录可观测告警但不改变已完成请求的授权结果。

### 9. 管理 API、页面、`i18n`和插件生命周期闭环在插件内

`linapro-auth-aksk`采用`source`、`managed`、`tenant_aware`，显式启用`en-US`和`zh-CN`。插件在稳定宿主目录下声明管理菜单与客户端、密钥、策略的查看、新增、修改、删除和授权按钮权限。

所有管理接口挂载在插件命名空间，使用REST资源语义并只接受`JWT`主体。创建密钥响应是唯一允许包含明文`SK`的响应；列表、详情和策略投影只返回`AK`显示信息、是否配置、状态和时间戳。页面通过一次性结果弹窗提供复制操作，弹窗关闭后不再恢复`SK`。

插件禁用时管理菜单、页面、路由和`HMAC`提供者全部隐藏或停止生效，数据不删除；重新启用后恢复。插件卸载按现有插件生命周期询问或接收是否清理数据的明确选项，不在普通禁用路径破坏密钥和策略。

### 10. `OpenAPI`和审计必须理解机器主体

宿主`OpenAPI`增加`LinaHMAC`安全方案，并按接口`actors`生成`BearerAuth`、`LinaHMAC`或二者择一的操作安全声明。接口文档同时投影稳定操作码、资源码和`read/write`动作，插件启用`i18n`时维护对应接口文档翻译资源。

统一业务上下文和操作日志增加`actorKind`、`subjectId`、`credentialId`或等价机器审计字段。对外展示的`credentialId`只保留可识别的`AK`前缀或后缀。操作日志必须同时清理请求头、请求参数、响应结果和错误文本中的`Authorization`、签名、`SK`、密文、主密钥和一次性创建响应；明文`SK`不得因响应截断逻辑而被部分保留。

### 11. 插件提供机器专用读写验证资源

`linapro-auth-aksk`提供同一资源类型下的`GET`与`PUT`机器专用验证接口，作为插件拥有的稳定测试资源。两个接口分别声明唯一`operation`、同一`resource`、`read`或`write`动作及`actors:"machine"`，从而让授权目录、策略管理和实际请求使用完全相同的生产认证与授权链，而不是前端模拟或绕过中间件的测试入口。

验证资源不读写数据库、不修改机器客户端、密钥或策略，也不承载可复用业务状态。读取响应只投影宿主认证链注入的机器主体类型、客户端标识、脱敏凭证前后缀和可信租户；写入响应在同一投影上回显有界测试载荷。接口不得返回完整`AK`、`SK`、签名、`nonce`、主密钥、密文或请求头。

该资源只允许机器主体访问。`Bearer JWT`管理员、无签名请求、未绑定策略、只具备接口或资源单侧权限的请求均失败关闭。完整授权后读写分别成功；同一`nonce`重放、策略解绑、策略停用、密钥停用或客户端停用后再次请求必须拒绝。资源没有持久数据，因此不存在数据权限查询或`N+1`；租户边界直接来自可信机器主体且不能被请求载荷覆盖。

## Risks / Trade-offs

- **[通用认证接缝扩大宿主安全面]** → 认证方案唯一注册、显式主体类型、机器接口默认拒绝、启动审计和失败关闭共同限制扩展面。
- **[数据库泄漏可能暴露加密密文]** → 主密钥与数据库分离、`AES-GCM`关联数据绑定、版本化密钥轮换，并禁止缓存明文。
- **[应用进程被攻破仍可使用主密钥]** → 这是对称`HMAC`协议的固有限制；通过最小权限部署、密钥环轮换、审计和快速撤销降低风险，后续可独立增加外部`KMS`。
- **[分布式协调故障导致机器请求不可用]** → 重放和撤销安全优先于可用性，机器认证失败关闭；提供明确指标、日志和恢复检查。
- **[接口元数据维护遗漏]** → 机器访问默认拒绝，并通过静态路由、源码插件和动态插件治理测试阻断缺失或重复元数据。
- **[一次性`SK`被通用响应日志记录]** → 在创建接口、统一审计和错误链路实施多层脱敏，并加入针对完整及截断响应的自动化测试。
- **[策略数量增长影响认证性能]** → 认证时一次性装配集合快照、设置策略和操作数量上限，并通过查询次数断言或集成测试验证无`N+1`。
- **[插件禁用造成外部集成中断]** → 管理页面明确提示插件停用会关闭所有机器访问，禁用保留数据且重新启用可恢复。

## Migration Plan

1. 先在宿主增加通用主体、认证提供者注册、机器上下文、路由元数据审计和`OpenAPI`投影，保持所有现有接口默认只允许用户主体。
2. 增加`linapro-auth-aksk`插件结构、SQL、生成代码、服务、管理 API、页面和`i18n`资源；没有有效主密钥配置时认证提供者和新访问密钥创建失败关闭，但宿主与`Bearer JWT`管理面继续启动。
3. 接入`HMAC`验证、策略快照、协调修订号、`nonce`去重和机器审计，完成单机及集群验证。
4. 仅对本次明确选择的非用户级资源接口添加`machine`元数据；未迁移接口继续保持机器默认拒绝。
5. 发布前执行插件安装、启用、禁用、重新启用和卸载演练，以及密钥创建、轮换、撤销、过期、策略变更和跨节点重放测试。

回滚时先禁用`linapro-auth-aksk`，所有机器认证立即失败关闭，`Bearer JWT`保持可用；随后回滚接口机器元数据和宿主提供者接缝。插件数据默认保留，只有显式卸载清理才删除。
