# 数据权限规则

## 适用范围

本规则约束所有新增或修改的数据操作接口、服务方法、插件宿主服务适配器、源码插件后端数据访问、聚合统计、下载读取、授权关系和执行类动作的数据权限接入。

## 强制遵守场景

以下任一场景命中时，必须先读取并遵守本规则：

- 新增或修改列表查询、导出、详情、批量信息、聚合统计、下载或读取内容
- 新增或修改创建后的可见性边界、更新、状态变更、删除、批量删除或授权关系变更
- 新增或修改执行类动作、任务触发、插件通过宿主发布服务访问数据的路径
- 修改任何可能暴露范围外数据存在性、候选项、数量、关联关系或可见性边界的逻辑

## 总体要求

- 所有新增或修改的数据操作接口都必须显式评估并接入角色管理中的数据权限控制。
- 参考已接入数据权限的用户管理、角色授权用户、文件管理、定时任务/任务日志、在线用户等模块实现方式。
- 读取类接口必须在数据库查询阶段注入数据权限过滤，避免先查出范围外数据再在内存中过滤。
- 详情、下载、写操作和执行类动作必须在操作前校验目标记录可见性。
- 聚合统计不得泄露范围外数据存在性。
- 插件通过宿主发布服务访问数据时，必须遵守与宿主 API 等价的数据权限边界。

## 读取类接口要求

- 列表、搜索、导出、树形数据、下拉选项、候选用户列表和批量信息必须在查询条件中注入数据权限。
- 查询条件不得先返回全量数据再在内存中过滤，除非 OpenSpec 明确说明数据规模、安全边界和例外原因。
- 间接关联查询必须确认不会通过关联名称、数量、排序、分页总数或候选项泄露范围外数据。
- 组织、部门、岗位等能力禁用时，数据权限策略必须按规范降级，不得因缺少组织能力直接放开全量数据。

## 写操作和执行动作要求

- 创建操作必须说明创建后记录对当前用户和其他角色的可见性边界。
- 更新、状态变更、删除、批量删除、授权关系变更和执行类动作必须在操作前校验目标记录是否在当前用户数据权限范围内。
- 批量操作必须在任一目标不可见时整体拒绝，或在规范中明确逐项处理语义、响应结构和审计方式。
- 执行动作不得通过任务 ID、文件 ID、插件 ID 或其他间接资源绕过目标数据可见性。

## 例外要求

确有业务例外时，必须在 OpenSpec 设计、任务和审查结论中说明：

- 权威边界
- 拒绝策略
- 为什么可以绕过常规数据权限
- 是否存在公开资源、当前用户自隔离消息、内置系统任务投影等合理例外
- 对应测试覆盖或治理验证

禁止未说明原因地绕过数据权限。

## 测试与验证要求

- 新增或扩大数据操作面时，必须新增或更新单元测试、集成测试或 E2E，覆盖有权限、无权限和边界条件。
- 读取类接口应验证数据库查询阶段已注入数据权限过滤。
- 详情和写操作应验证不可见目标被拒绝。
- 聚合统计应验证不会泄露范围外数据数量或存在性。
- 纯治理类变更可使用 OpenSpec 校验、静态检索、文件检查或审查结论作为验证证据。

## 审查要求

- 审查必须识别变更是否涉及数据操作；确认无数据操作影响时必须在审查结论中明确记录。
- 审查必须确认读取类接口在数据库查询阶段过滤。
- 审查必须确认详情、下载、写操作和执行类动作在操作前校验目标可见性。
- 审查必须确认业务例外已在设计、任务和审查结论中说明并具备验证证据。
