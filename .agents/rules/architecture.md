# 架构设计规则

## 适用范围

本规则约束项目定位、核心宿主边界、模块设计、字典枚举、模块启停、前后端解耦、业务能力可禁用和通用能力复用方式。

## 强制遵守场景

以下任一场景命中时，必须先读取并遵守本规则：

- 新增或修改业务模块、模块能力边界、模块启停或模块间依赖
- 新增或修改具有枚举语义的业务值、字典类型、字典数据或模块运行配置
- 调整默认工作台、系统管理、用户权限等内建能力与框架宿主边界
- 设计前后端模块降级、隐藏、恢复或按需启用逻辑
- 修改`apps/lina-core`通用服务契约、领域模型、组件能力或插件扩展边界

## 项目定位要求

- 项目统一定位是`面向可持续交付的 AI 原生全栈框架`。所有功能设计都必须围绕这一定位展开。
- 默认管理工作台、系统管理模块、用户权限模块等能力属于`LinaPro`提供的默认入口和内建通用能力，不构成项目的唯一产品边界。

## 核心宿主边界要求

- `apps/lina-core`是全栈开发框架的核心宿主服务，负责提供通用模块接口能力、组件能力、系统治理能力与插件扩展能力。
- `apps/lina-core`的设计必须优先保证通用性、稳定性和可复用性，不得与具体管理工作台页面的展示结构、交互细节或前端框架实现强绑定。
- 若需求仅来源于表格列、筛选项、树选择器、路由装配、工作台聚合、下拉选项等工作台展示变化，应优先通过工作台适配接口或前端适配层解决，而不是直接修改`lina-core`核心领域契约、通用`service`语义或存储模型。

## 模块功能设计要求

- 业务模块的枚举值都应当使用字典模块维护其对应的字典类型和字典数据，而不是在代码中硬编码枚举值。
- 操作日志中的操作类型、操作结果，登录日志中的登录状态，文件管理的业务场景等都应该使用字典类型维护。
- 具有状态、类型、阶段、动作、执行模式、排序方向、过滤操作符等枚举语义的后端值，还必须遵守`.agents/rules/backend-go.md`中的命名类型和常量要求。
- 涉及用户可见字典文案时，还必须读取`.agents/rules/i18n.md`。

## 模块解耦设计原则

- 所有前后端模块必须采用解耦设计，业务模块支持按需启用和禁用。
- 每个业务模块应当独立，可以通过配置禁用。
- 禁用某模块后，所有依赖该模块的功能必须自动降级或隐藏，不能出现报错或空白区域。
- 当一个模块被禁用时，前端所有涉及该模块的 UI 元素必须完全隐藏，而非仅禁用或置灰。范围包括菜单项、表单字段、表格列、搜索条件、按钮等。
- 后端服务间的依赖应通过接口或可选引用实现，避免硬依赖。
- 当被依赖模块禁用时，相关字段返回零值或忽略即可，不应抛出错误，除非规范明确要求拒绝访问。
- 模块禁用仅影响功能和展示层，不应删除或破坏已有数据。
- 重新启用模块后，历史数据应能正常恢复使用。

## 设计记录要求

- OpenSpec 设计必须说明新增或修改模块是否属于宿主通用能力、源码插件能力、工作台适配层能力或前端展示适配。
- 涉及模块启停时，设计必须说明禁用后的前端隐藏策略、后端降级策略、数据保留策略和重新启用恢复路径。
- 涉及枚举语义时，设计必须说明字典类型、字典数据来源、运行时展示方式和`i18n`影响。

## 验证要求

- 涉及模块启停或前端隐藏时，必须使用单元测试、E2E 或治理验证覆盖启用和禁用两种状态。
- 涉及后端模块降级时，必须验证依赖模块禁用后不会出现 500、空白页或不可恢复错误。
- 纯治理文档变更可以使用`openspec validate`、静态检索、文件检查或审查结论作为验证证据。

## 审查要求

- 审查必须确认变更没有因工作台展示需求直接污染`lina-core`核心领域契约。
- 审查必须确认模块启停、前端隐藏、后端降级和数据完整性策略已记录并验证。
- 审查必须确认具有枚举语义的业务值没有绕过字典治理。
- 涉及数据操作、缓存、插件或`i18n`时，审查必须继续读取对应规则文件。
